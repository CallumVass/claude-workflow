# Stitch — UI Design Reference

Stitch is the source of truth for UI design. It provides two things:

1. **DESIGN.md** — a design system document (auto-generated by Stitch) in the project root. Contains colors, typography, components, spacing, do's/don'ts, and the creative direction. This is your styling bible.
2. **Screens** — HTML mockups stored in a Stitch project. Each screen is a pixel-perfect reference for a specific page or component.

## How Stitch Connects to the Project

- **DESIGN.md** lives in the project root. If it doesn't exist, Stitch is not in use — skip all Stitch workflows.
- **Stitch project ID** is specified in the PRD or issue (not in DESIGN.md). Look for a `stitch_project_id` field or similar reference.

## Workflow: Implementing UI

### 1. Read the Design System

Read `DESIGN.md` first. Every visual decision (colors, fonts, spacing, elevation, component patterns) must follow its rules. Do not deviate.

### 2. Discover Available Screens

Call `list_screens` with the Stitch project ID to see what screens exist. Match screens to the components/pages you need to implement.

### 3. Fetch Screen HTML

For each screen relevant to your work:

1. Call `get_screen` with the project and screen IDs.
2. Extract the HTML — this is your **exact visual target**.
3. Implement the component to match the HTML structure, styling, and layout. Adapt to the project's framework (React, Vue, etc.) but the visual output must match.

### 4. Generate Missing Screens

When implementing a UI component that has **no matching screen** in Stitch:

1. Call `generate_screen_from_text` with:
   - `projectId`: the Stitch project ID
   - `prompt`: describe the component, its purpose, content, and reference the design system from DESIGN.md for consistency
   - `deviceType`: match the project's target (DESKTOP, MOBILE, etc.)
2. Fetch the generated screen with `get_screen` to get the HTML.
3. Implement the component matching the generated HTML.

### 5. Edit Existing Screens

If a screen needs changes (requirements changed, new section added):

1. Call `edit_screens` with the screen IDs and a prompt describing the change.
2. Re-fetch with `get_screen` to get updated HTML.
3. Update the implementation to match.

## Rules

- **DESIGN.md is the styling authority.** All visual decisions (colors, spacing, typography, elevation, component patterns, do's/don'ts) come from DESIGN.md.
- **Screen HTML is the layout authority.** Translate the HTML into the project's framework idioms (JSX, Vue SFC, etc.), but the visual output must match exactly.
- **Don't deviate from the design.** If the design conflicts with requirements, flag it — don't silently "improve" it.
- **Generation is slow.** `generate_screen_from_text` can take minutes. Don't retry on timeout — check with `get_screen` later.
